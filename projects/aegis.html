Aegis Communication Protocol

A resilient, low-latency state synchronization protocol for dual-computer robotics platforms.

C++20 Linux Sockets Jetson Orin

Project Overview

The Aegis protocol was developed to provide resilient communication and rapid failover between redundant flight computers in autonomous robotics systems. The platform uses two onboard computers—a Jetson Orin Nano acting as the primary autonomy computer and a Jetson Nano serving as a hot standby controller. If the primary system experiences a crash, kernel hang, or ROS2 node failure, the backup computer must assume control of the drivetrain immediately to prevent loss of control or collision.

Aegis provides deterministic communication between the two computers using a lightweight UDP protocol optimized for low latency and minimal overhead. The system continuously synchronizes robot state, health metrics, and control authority so that the backup computer can take control within milliseconds without requiring system reinitialization.

System Architecture

The protocol operates across a dedicated internal Ethernet network connecting the two onboard computers and the robot's control systems. The primary flight computer performs perception, navigation, and planning while continuously publishing state updates to the backup computer. These updates include drivetrain commands, actuator states, sensor summaries, and fault conditions.

The backup computer mirrors the operational state of the robot but does not transmit commands unless a failure condition is detected. If the primary computer stops responding to heartbeat messages or reports a critical fault, control authority automatically transfers to the backup system. This architecture allows the robot to continue operating even if the primary autonomy stack fails.

Key Features

  • ▹
    High-Frequency Heartbeat Monitoring: Continuous health monitoring at up to 100Hz allows the backup computer to detect system failure in under 10 milliseconds.
  • ▹
    Deterministic Failover: The backup computer maintains synchronized drivetrain and actuator state so that it can immediately assume control without requiring ROS2 node startup or hardware reinitialization.
  • ▹
    Fault Detection and Reporting: The protocol supports structured fault codes that allow subsystems to report hardware failures, communication errors, or safety violations.
  • ▹
    Exclusive Control Authority: AEGIS prevents split-brain behavior by dynamically enforcing which onboard computer is permitted to transmit motor commands, ensuring that only one controller can drive any actuator at a time.

Protocol Design

Aegis messages are transmitted over UDP using a lightweight binary packet format. Each packet contains a message identifier, timestamp, and compact payload representing a specific subsystem update. Message IDs categorize traffic into telemetry, configuration updates, fault reporting, and synchronization commands.

The protocol avoids the overhead of traditional middleware while still supporting structured communication between subsystems. This design allows the robotics platform to maintain extremely low latency even when operating under heavy sensor load or high-frequency control loops.

Real-World Application

Aegis was designed for a lunar excavation rover competing in NASA's Lunabotics competition. The rover must operate autonomously in a harsh environment where system crashes or software faults could cause the robot to collide with obstacles or lose control of its drivetrain. By providing fast failover and continuous state synchronization between computers, Aegis ensures the robot remains controllable even during unexpected failures.
